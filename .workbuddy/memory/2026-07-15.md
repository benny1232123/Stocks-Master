# 2026-07-15 工作日志

## daily_backtest.py 性能优化 + 进度输出
**问题**：GitHub Actions 回测步骤 48min 仍 59%，频繁超时。根因：30信号日×100只/日内联过滤=3000+次 akshare HTTP（海外慢），循环内零进度。
**修复**：
1. 新增 `_collect_all_candidate_codes()` 预收集全量标的，`main()` 阶段1 每只股票仅拉一次最宽区间 K 线（填满进程缓存+文件缓存），后续 `_compute_boll_levels`/引擎全部命中缓存不再触发 HTTP。
2. 内联过滤循环加 `[过滤] N/M ...` 每20条进度；预拉阶段加 `[预拉 N/M] (P%) code ...` 每10条；逐日回测加 `(i/total)` 序号和总耗时统计。
3. 所有 `print` 加 `flush=True` 保证 GitHub Actions 实时流式输出（不被 Python 缓冲吞掉）。

## 二次优化（2026-07-16，CI 实跑验证后）
**CI 实跑结果**：首次优化后预拉阶段显示 `[预拉 760/819]`——去重后仍有 **819 只** × 海外 ~4s = **55min 刚好超时**。CCTV push 因文件不存在 exit 1。
**二次修复 (4 项)**：
1. **strategy_cache.py**：push 文件不存在时 `return True`（无需上传≠错误），CCTV 个股池可能为空是正常分支。
2. **跳过已完成回测**：新增 `_skip_completed()` —— Multi-Backtest-{date}-summary.csv 已存在且窗口走完的信号日直接跳过，不纳入预拉集合。
3. **并发预拉**：`ThreadPoolExecutor(max_workers=4)` 替代串行循环，~4x 提速预拉阶段。
4. **默认 LOOKBACK_DAYS 从30降到14**（workflow + 代码默认同步），减少待回测信号日数；已完成的历史结果自动保留不被覆盖。

**预期效果（二次优化后）**：
- 首次运行（无历史缓存）：14天×40只≈200~300 去重 → 4线程预拉 ≈ 3-5min
- 后续运行（大部分已完成）：可能仅需回测 1~3 天新信号 → 预拉 <1min
