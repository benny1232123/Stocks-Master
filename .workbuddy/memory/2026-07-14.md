## 回测（Multi-Backtest）静默失败根因与修复（commit ffa1026，已推送）
- **现象**：Daily-Action-List 有（07/13 在），但 Multi-Backtest 停在 07/10；CI 只有一个 `daily-pick.yml`，其中回测步骤标了 `continue-on-error: true`。
- **根因**：`daily-pick.yml` 的 commit 步骤只 `git add -f stock_data/daily_cache/`，**漏掉 `stock_data/cache/`（K 线历史缓存目录）**。CI 每次 fresh checkout 都无缓存 → 回测步骤对近 30 天×~30 只票每只抓 3 次历史 K 线（内联过滤 + 波动率止损 + 引擎各一次）≈2700 次抓取 → 40 分钟**超时** → `continue-on-error` 静默吞掉 → 只提交了日报，回测文件全无。
- **关键佐证**：本地用 baostock 后端 + 仅 07/13（LOOKBACK_DAYS=1）跑通，证明我改的引擎逻辑（capital_scale/stop_pct）没问题；fetch_daily_k 三级回退（tdx→akshare→baostock）catch 异常不抛，排除"异常崩溃"。真正差异是 CI 用 akshare + 30 天窗口 + 空缓存 → 超时。
- **修复**：
  1. `daily_backtest.py` 注入**进程内 K 线缓存**（一次运行跨信号日/跨调用对同一只票只抓一次）；市场仪表盘只算一次由 `main()` 传入；单只 K 线失败容错返回 {} 放行；每个信号日回测包 try/except 异常跳过；写 `backtest_status.txt` 让失败可见。
  2. `daily-pick.yml`：commit/artifact 增加 `stock_data/cache/` 与 `backtest_status.txt` 持久化；回测超时 40→55 分钟；步骤结束 `cat backtest_status.txt` 打印状态。
- **验证**：本地 baostock 跑通，status 文件正确（run_date/generated/skipped/range）。
- **恢复预期**：下次 daily-pick（或手动 workflow_dispatch）缓存持久化 + 入内存缓存 → 回测在时限内跑完并 commit；滚动窗口每天重算所有 eligible 信号日，07/13 等会随交易日推移逐步延长持有期。本地已后台跑全窗口回测以立即补数据。
- **教训**：CI 里「会写文件但只在步骤末尾 git add」的步骤，若中途超时则产物永不提交；缓存类产物必须显式 `git add` 持久化，否则每轮重复昂贵抓取。

## 修复云端白屏 bug（commit a58a271，已推送，22:12）
- **现象**：用户反馈"一进去就白屏了"，F12 截图显示 `TypeError: Cannot read properties of null (reading 'toFixed')` 在 `Array.map` 内（index-C1a8l_gw.js:40）。
- **根因**：上次修 NaN 时将 `num()` 改为返回 `null`（而非 `NaN`），但 JS 的 `isNaN(null)===false` 导致 `!isNaN(null)===true`——null 值通过了 `!isNaN()` 守卫直接调用 `.toFixed()` → TypeError → React 组件树在 `.map()` 内崩溃 → 纯白屏。崩在 DailyExpandableList 组件的 score/buyPrice/posPct/amt 共 7 个 `.toFixed()` 调用点。
- **修复**：新增 `hasNum(v)` 辅助函数（`v!=null && !isNaN(v)`），替换组件中所有 `!isNaN(varName)` 守卫为 `hasNum(varName)`；构建验证通过（1792 模块，JS hash 从 C1a8l→CIY8Wb 确认变更打入）；推送 `a58a271` 触发 Render 自动重新部署。
- **教训**：JS 的 `isNaN/null/undefined` 语义坑极多；`num()` 返回 null 时所有下游必须用 `hasNum()` 或 `?.` 可选链，不能依赖 `!isNaN()`。
