## daily_backtest.py 去除多线程预拉（用户要求）
**背景**：上一轮二次优化用了 `ThreadPoolExecutor(max_workers=4)` 并发预拉 K 线来提速，但用户反馈「尽量不要用多线程，容易搞崩接口（akshare 上游）」。
**改动**：
1. `scripts/daily_backtest.py`：删除 `from concurrent.futures import ThreadPoolExecutor, as_completed` 和 `from threading import Lock`；预拉阶段改回串行 `for` 循环。
2. 新增 `PREPULL_INTERVAL` 环境变量（默认 0.3s），每只成功/失败后 `time.sleep(interval)`，给上游接口喘息，避免打崩。
3. 进度输出增加 **ETA**（预计剩余时间），更易判断是否会超时。
4. `.github/workflows/daily-pick.yml`：移除 `PREPULL_WORKERS: "4"`，新增 `PREPULL_INTERVAL: "0.3"`。
**补偿速度的保留措施**（依然有效）：`_skip_completed()`（已完成回测的信号日直接跳过，日常增量只需预拉 1~3 天新信号）+ `LOOKBACK_DAYS` 默认 14（减少候选集）。冷启动全量预估 14天×去重约300只×~4s ≈ 20-25min，日常增量 <5min，均在 55min 超时内。

## CI 提交步骤 exit 128 根因修复（治本）
**第一层（无效）**：先用 `git add -f --ignore-missing`，但 GitHub Runner 的 Git 版本要求 `--ignore-missing` 必须搭配 `--dry-run` → 仍 fatal。改回 `[ -d/-f 路径 ] && git add -f ... || true`。
**用户的「治标不治本」点醒 + 真正根因**：
- `stock_data/cache/` 在 `.gitignore` 第 40 行被**显式忽略**（本地临时态 `realtime_snapshot.pkl`；`strategy_cache.py` 走 Supabase 云端，不用此目录）。CI 全新 checkout 时它物理上不存在。
- workflow 却 `git add -f stock_data/cache/` → pathspec 报错 → **整步 abort** → 连本该提交的 `daily_cache/`、`k_data/`、`Daily-Action-List`、`Multi-Backtest` 全都没提交。
- 致命连锁：因为 `k_data/*.csv` 从未被成功提交，每次 CI 都是**冷启动**——前面「文件缓存暖启动 + 跳过已完成」的回测优化等于白做，这才是回测长期慢的隐藏根因。
**治本修复**：从 `git add` 列表**移除 `stock_data/cache/`**（故意忽略就不该提交）；保留 `daily_cache/`、`k_data/`（已 git 跟踪、需跨 run 持久化以暖启动）；每个 add 前先判路径存在（`[ -d/-f ] && ... || true`），杜绝此类 bug 复发。YAML 校验通过。
